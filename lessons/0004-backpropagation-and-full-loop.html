Lesson 0004: Backpropagation and the Full Loop

Completing the four-phase cycle — how rewards flow up the tree and the engine takes shape.

1. What Backpropagation Does

After the Simulation phase returns a reward, we need to update every node that was visited during this MCTS iteration. This is Backpropagation (also called backup). Each node along the path gets two updates:

  • $n(s) \mathrel{+}= 1$ — increment visit count
  • $w(s) \mathrel{+}= \text{reward}$ — add the simulation result

The reward travels up from the leaf (the node that was just expanded and simulated) all the way back to the root. Every node on that path learns from the outcome.

2. The Reward Sign Problem

In a two-player zero-sum game, what's good for the current player is bad for the opponent. If we simulate from a node where it's White's turn and White wins, the reward is $+1$ for that node. But from the parent node (where it was Black's turn), a White win is a loss — so the reward should be negated.

There are two common approaches:

2.1 Alternating Sign (Standard)

Negate the reward at each level as you ascend. This is the most principled approach for two-player games.

def backpropagate(node, reward): """Walk up the tree, updating stats. Negate reward at each level.""" while node is not None: node.visits += 1 node.wins += reward node = node.parent reward = -reward # flip for the other player
⚠️ Zero-sum assumption: This negation works for zero-sum games where a win for one player is a loss for the other. For games with draws, the reward is $+0.5$ for both sides — it passes through negation as $-0.5$ at the parent, which is wrong. Fix: only negate when the outcome is a decisive win ($\pm 1$), or store the reward from a fixed perspective (see below).

2.2 Fixed-Perspective Reward (Simpler)

Store every node's reward from White's perspective (or Player 0's perspective). During backpropagation, don't negate — but the simulate() function must return a reward from White's perspective regardless of whose turn it is at the leaf.

def backpropagate(node, reward): """Walk up the tree, updating stats. No sign flip needed. Reward must already be from a fixed global perspective.""" while node is not None: node.visits += 1 node.wins += reward node = node.parent
💡 Which to choose? The alternating-sign approach matches the theory (each node tracks wins for the player who moved to it). The fixed-perspective approach is simpler to implement because there's no logic for draws vs. wins. Choose alternating-sign for clarity and correctness; choose fixed-perspective for simplicity. Both work in practice.

3. The Complete Iteration (All Four Phases)

import math import random def mcts_iteration(root, Cp=1.414): # 1. SELECTION — descend to a frontier or terminal node node = tree_policy(root, Cp) # 2. EXPANSION — if not terminal, add one new child if not node.is_terminal: node = expand(node) # 3. SIMULATION — rollout from the new/terminal node reward = simulate(node) # 4. BACKPROPAGATION — update stats up the tree backpropagate(node, reward)

And the top-level search function that ties everything together:

def mcts_search(root_state, iterations=1000, Cp=1.414): """Run MCTS from root_state, return the best action.""" root = MCTSNode(root_state) for _ in range(iterations): mcts_iteration(root, Cp) best = best_move(root) return best.state.last_action

4. Full Trace Through One Iteration

Let's watch one complete cycle on a tiny tree to see every phase in action:

Initial tree: Root (N=10, W=7) has children A (n=4, w=3) and B (n=6, w=4).
Root still has one untried move C.
// Phase 1: tree_policy(root) // root is not fully expanded (C untried) → returns root for expansion // Phase 2: expand(root) // pops C, applies it, creates child C (state_C, parent=root) root.children = [A, B, C] // Phase 3: simulate(C) // random rollout from state_C ... White wins → reward = +1 // Phase 4: backpropagate(C, +1) // C: visits=1, wins=1 (White's turn reward += +1) // root: visits=11, wins=8 (Black's turn reward += -1 ... wait, negated!)
💡 Sign walkthrough: If C was reached on White's turn and White won (reward $+1$), backpropagating with alternating sign gives C $+1$ (good for White), but root (which was Black's turn when White moved to C) gets $-1$ (good for Black means White's win is bad). Root's wins decrease, which is correct — it was a bad move for Black.

5. The Complete Engine Contract

Here's the full interface your game-specific state object must implement:

MethodReturnsDescription
legal_moves()listAll legal moves from this state
is_terminal()boolTrue if the game is over
apply(move)StateReturn a new state after making move
clone()StateReturn an independent copy
reward()floatTerminal reward from current player's perspective
last_actionmoveProperty storing the move that led to this state
🧠 Design reflection: With these six methods, MCTS is completely game-agnostic. The same mcts_search function works for Chess, Checkers, Go, Tic-Tac-Toe, and even non-game tree-search problems. You swap in a new state object and the algorithm adapts automatically. This is the power of the abstraction.

6. Check Your Understanding

✓ Quiz: Backpropagation and the Full Loop

1. What two statistics does backpropagation update at each node?

Correct! Each visited node increments its visit count $n$ and adds the (possibly negated) reward to its total $w$.

2. In alternating-sign backpropagation for a two-player game, what happens to the reward at each level?

Correct! What's a win for the current player is a loss for the opponent at the parent level, so the sign flips at each step up the tree.

3. How many game-specific methods does the state object need to implement?

Correct! The MCTS engine is completely agnostic to the game — it only depends on these six methods. Everything else is generic.

4. What path does backpropagation follow through the tree?

Correct! Backpropagation walks the exact path that Selection followed, in reverse — from leaf back to root.

7. Summary

  • Backpropagation walks from leaf to root, incrementing $n$ and adding $w$ at each node.
  • Reward sign alternates at each level for two-player zero-sum games.
  • Draws ($+0.5$) need special handling with alternating sign (only negate $\pm 1$).
  • The complete engine needs exactly six game-specific methods.
  • mcts_search is the top-level function: build root, loop iterations, return best action.
