Lesson 0006: Memory, Parallelism, and Advanced Tradeoffs

Taking your MCTS engine from a proof-of-concept to a production-quality system.

1. Tree Reuse Across Moves

After mcts_search returns the best action, the opponent makes their move. We could discard the entire tree and rebuild from scratch — but that's wasteful. The subtree rooted at the opponent's chosen move is still valid.

def advance_root(tree, opponent_move): """After the opponent plays, reuse the relevant subtree. Returns the child node that matches the opponent's move.""" for child in tree.children: if child.state.last_action == opponent_move: # Detach from parent, making this node the new root child.parent = None return child return None # no match; caller should rebuild
🧠 Why tree reuse matters: Imagine you ran 100,000 iterations to analyze the current position. After the opponent moves, the relevant subtree might already contain 5,000–20,000 iterations of data. By reusing it, you don't start from scratch — you keep that analysis and continue deepening it. This is one of the biggest performance wins in any MCTS engine.

2. Memory Management

Each MCTS node holds references to its state, parent, and children. In a long-running game with deep trees, memory can become a concern. Here are strategies:

StrategyHow It WorksWhen to Use
No pruningKeep every node in memoryPrototyping, games with <100K nodes
Threshold pruningKeep only nodes with visits $> k$High-branching games (Go, large Chess)
Virtual lossArtificially inflate visit count to discourage threadsParallel search (covered below)
Transposition tableHash states to nodes; merge equivalent positionsGames with many transpositions (Chess, Checkers)

2.1 Transposition Tables

In many games, the same board position can be reached through different move sequences (transpositions). A transposition table hashes the game state and merges nodes that represent the same position. This collapses the tree and avoids redundant computation.

class MCTSNode: def __init__(self, state, parent=None, transposition_table=None): self.state = state self.parent = parent self.children = [] self._untried_moves = None self.visits = 0 self.wins = 0 if transposition_table is not None: h = hash(state) if h in transposition_table: # Merge statistics from the existing equivalent node existing = transposition_table[h] self.visits = existing.visits self.wins = existing.wins self.children = existing.children else: transposition_table[h] = self
⚠️ Caution: Transposition tables add complexity. The hash function must be fast and collision-free (or handle collisions). For games with no transpositions (like Go, where every stone placement is unique), they add overhead for no benefit.

3. Parallel MCTS

MCTS is embarrassingly parallel — you can run many iterations simultaneously. But there's a catch: if two threads both try to update the same node's statistics at the same time, you get race conditions.

3.1 Root Parallelism (Simple)

Run $N$ independent MCTS searches from the same root, each on its own thread. After all finish, average their visit counts. No shared state, no locks — but each thread wastes time rebuilding the same tree.

from concurrent.futures import ThreadPoolExecutor def parallel_mcts_search(root_state, total_iterations, num_threads=4, Cp=1.414): iters_per_thread = total_iterations // num_threads roots = [] with ThreadPoolExecutor(max_workers=num_threads) as executor: futures = [] for _ in range(num_threads): root = MCTSNode(root_state) roots.append(root) futures.append(executor.submit(_run_search, root, iters_per_thread, Cp)) # Average visit counts across all independent trees combined = MCTSNode(root_state) for r in roots: for child in r.children: # find corresponding child in combined and merge stats merge_into(child, combined) return best_move(combined)

3.2 Tree Parallelism (Advanced)

Share a single tree across all threads. Each iteration still follows the four phases, but threads coordinate on shared node statistics. This is more memory-efficient but requires virtual loss: before descending, a thread increments a node's visit count (artificially) as a "lock" to discourage other threads from following the same path.

def uct_select_child_parallel(node, Cp): """Like uct_select_child, but treats virtual losses as extra visits. A thread currently searching this path has already incremented node.visits (virtual loss), making it less attractive to others.""" best_score = -float('inf') best_child = None for child in node.children: if child.visits == 0: return child exploit = child.wins / child.visits explore = Cp * math.sqrt(math.log(node.visits) / child.visits) score = exploit + explore if score > best_score: best_score = score best_child = child return best_child

MemoryHigh (each thread builds its own tree)Efficient (one shared tree)
Speedup (4 cores)~3.5x (some wasted work)~3.8x (near-linear)

4. Rollout Policy Engineering

Earlier we said uniform random is "good enough." But when you need to squeeze out more performance, the rollout policy is where you spend your effort:

  • Heuristic bias: Weight moves by domain knowledge (e.g., capture checks in Chess, attack weak territories in Risk) rather than choosing uniformly.
  • Cutoff rollouts: Stop the rollout early and use a heuristic evaluation function. This trades accuracy for speed — useful when reaching a terminal state takes many moves.
  • MAST (Move-Average Sampling Technique): Track a global table of "moves that win." During rollouts, pick moves with historically good global win rates.
# Heuristic rollout for Checkers: def heuristic_rollout(state): while not state.is_terminal(): moves = state.legal_moves() # Prefer captures; otherwise random captures = [m for m in moves if m.is_capture] if captures: move = random.choice(captures) else: # Weighted random: prefer forward moves slightly weights = [2.0 if m.is_forward else 1.0 for m in moves] move = random.choices(moves, weights=weights)[0] state.apply(move) return state.reward()

5. The Complete Engine Architecture

MCTS Engine Architecture

mcts_search() → main entry point
  ┗ mcts_iteration() → one four-phase cycle
    ┗ tree_policy() → Selection (UCT descent)
      ┗ uct_select_child() → UCT formula
    ┗ expand() → adds one child node
    ┗ simulate() → rollout
    ┗ backpropagate() → walk up, update stats
  ┗ best_move() → pick most-visited child
  ┗ advance_root() → tree reuse for next turn

All depend on MCTSNode and game-specific State.

6. Check Your Understanding

✓ Quiz: Advanced Tradeoffs

1. After the opponent makes a move, how can you avoid rebuilding the MCTS tree from scratch?

Correct! The subtree below the opponent's move is still valid. Detach it from its parent and make it the new root. This preserves thousands of iterations of analysis.

2. What is a transposition table used for?

Correct! Transpositions are common in Chess and Checkers. Hashing states and merging equivalent nodes saves computation and memory.

3. What is the simplest approach to parallel MCTS?

Correct! Root parallelism is trivial to implement (no locks, no shared state). It wastes some work (each thread rebuilds the tree) but gives good speedup for minimal code complexity.

4. What does virtual loss do in tree-parallel MCTS?

Correct! Virtual loss acts as a temporary "lock." A thread increments a node's visits before searching it, making it less attractive to other threads via UCT's exploration term.

7. Summary

  • Tree reuse via advance_root() is the single biggest performance win — never rebuild from scratch.
  • Transposition tables merge equivalent states; essential for Chess/Checkers, unnecessary for Go.
  • Root parallelism is the simplest way to parallelize: independent searches, averaged results.
  • Tree parallelism with virtual loss is more advanced but more memory-efficient.
  • Heuristic rollouts improve per-iteration quality at the cost of per-iteration speed.
  • The engine architecture has a clean separation: generic MCTS core + six-method game-specific State.
