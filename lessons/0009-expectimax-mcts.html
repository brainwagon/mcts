Lesson 0009: ExpectiMax MCTS for Stochastic Games

Why searching deeper can sometimes make you play worse, and how explicitly modeling chance nodes fixes it.

1. The Open-Loop Paradox

In Lesson 0008, we encountered a fascinating paradox in our Yahtzee engine:

  • The pure rollout heuristic (0-ply search) averaged ~191 points.
  • Running MCTS with that exact same heuristic for 500 iterations averaged ~156 points.

How can adding search make the agent perform worse? The culprit is our use of an Open-Loop tree architecture in a highly stochastic game.

In Open-Loop MCTS, tree nodes represent actions rather than states. A path through the tree looks like this:
Root → Keep(1,1) → Score(Ones) → Keep(2,2)

Because it only branches on actions, the tree assumes it can commit to a sequence of moves without knowing the outcome of the random dice rolls in between them. If a random simulation happens to roll lucky 2s on the third turn, the UCB formula artificially inflates the value of that entire blind sequence. The MCTS ends up overriding the sensible root heuristic in favor of rigid, hallucinated action sequences.

2. Game Features that Break Open-Loop MCTS

Open-Loop MCTS is not universally bad. It works brilliantly in games with hidden information (like choosing to draw a card in Blackjack before you know what the card is). However, it catastrophically fails in games that possess these two features:

  1. Fully Observable Chance Outcomes: In Yahtzee, the random event (rolling dice) happens, and the player gets to observe the outcome before making their next decision.
  2. State-Dependent Action Value: The optimal action depends entirely on the chance outcome. The decision to "Score Yahtzee" is objectively brilliant if you rolled five 6s, and objectively terrible if you rolled a small straight.
The Core Problem: Yahtzee is a Markov Decision Process (MDP). Open-Loop MCTS attempts to solve an MDP by guessing an optimal sequence of actions, rather than guessing an optimal policy (a mapping of states to actions).

3. The Solution: ExpectiMax MCTS (Closed-Loop)

To fix this, we must transition to a Closed-Loop MCTS (often implemented as ExpectiMax MCTS). In this architecture, the tree explicitly branches not just on the player's choices, but on the environment's random outcomes.

The tree alternate between two types of nodes:

  • Decision Nodes: It is the player's turn to act. The children of this node represent all legal Actions. The value of this node is determined by the UCB formula (maximizing the best action).
  • Chance Nodes: The player has taken a stochastic action (like `KEEP`), and the environment must respond. The children of this node represent all possible Random Outcomes (the newly rolled dice). The value of this node is the Expected Value (the weighted average) of its children.

Visualizing the Tree

[Decision Node] State: Dice [1, 2, 3, 4, 6], Rolls Left: 1 | +-- (Action: KEEP 1,2,3,4) --> [Chance Node] Keeping 4 dice, rolling 1 | +-- (Outcome: Roll 1) --> [Decision Node] State: [1,1,2,3,4] (1/6 prob) +-- (Outcome: Roll 2) --> [Decision Node] State: [1,2,2,3,4] (1/6 prob) +-- (Outcome: Roll 5) --> [Decision Node] State: [1,2,3,4,5] (1/6 prob)

4. Why ExpectiMax Fixes the Paradox

By adding Chance Nodes, the tree is no longer forced to commit to a blind sequence of actions.

When the MCTS traverses down the tree and chooses an action at a Decision Node, it hits a Chance Node. The simulation then randomly rolls the dice, and the tree branches down to a new Decision Node specific to that exact dice roll.

Because the child node represents the true game state, the MCTS can learn a highly specific, reactive policy: "If I take this KEEP action, and I happen to roll a 5, my next action will be to Score Large Straight. But if I roll a 1, my next action will be to Keep the 1s."

Implementation Details: In standard ExpectiMax MCTS, when traversing a Chance Node during the Selection phase, you don't use UCB. You simply sample an outcome based on the environment's probability distribution (e.g., rolling a random die). During Backpropagation, the Chance Node's value updates as the mathematical average of all simulations that passed through it.

By making the tree structurally aware of the environment's randomness, MCTS stops hallucinating rigid paths and instead learns flexible, reactive strategies, allowing deep searches to achieve true expert-level play.
