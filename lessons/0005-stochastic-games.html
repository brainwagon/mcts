Lesson 0005: Handling Stochastic Games

Dice rolls, shuffled decks, and other random elements in the game tree.

1. The Challenge of Randomness

So far we've assumed deterministic games: the same move from the same state always produces the same next state. But many games include random elements — Risk has dice rolls, Backgammon has dice, Poker has shuffled decks. After a player makes a move, randomness (not a player) determines the outcome.

MCTS handles this by introducing chance nodes into the tree — nodes where "nature" or "the dice" makes the decision instead of a player.

Deterministic tree:
Player move → Player move → Player move → Terminal

Stochastic tree:
Player move → Dice roll → Player move → Dice roll → Terminal

2. Modelling Chance Nodes

A chance node represents a random event with known probabilities. In Risk, when you attack $D$ defenders with $A$ attackers, the outcome is determined by dice rolls. The possible outcomes (how many armies each side loses) have computable probabilities.

We model this by making expand() and apply() aware of chance:

# Inside a Risk-style state: def apply(self, move): """Return a new state after making `move`. If the move involves randomness, return a ChanceState that stores the possible outcomes with their probabilities.""" if isinstance(move, AttackMove): outcomes = compute_attack_outcomes(self, move) return ChanceState(outcomes) else: return DeterministicState(self.board, self.player, move)
class ChanceState: """A state where randomness determines the next state(s).""" def __init__(self, outcomes): self.outcomes = outcomes # list of (probability, resulting_state) def legal_moves(self): """"Moves" are the possible random outcomes.""" return [i for i in range(len(self.outcomes))] def apply(self, outcome_index): """Resolve the chance event to a specific outcome.""" _, next_state = self.outcomes[outcome_index] return next_state def is_terminal(self): return False
💡 Note: The tree now interleaves player-decision nodes (where UCT selects) with chance nodes (where random outcomes are explored). When Selection reaches a chance node, it still descends into a child — but the "selection" among outcomes is not strategic; it's just ensuring all outcomes get sampled.

3. Two Approaches to Chance

3.1 Explicit Chance Nodes (Tree-Integrated)

Chance outcomes are stored as actual tree nodes. Selection descends into them like normal children. UCT is used for both player moves and chance outcomes — though the interpretation differs: for chance nodes, UCT ensures all outcomes are explored proportionally.

Player Decision NodeChance Node
Selection policyUCT (exploration vs. exploitation)UCT (ensures all outcomes sampled)
Visits $n$How often this move was chosenHow often this outcome was sampled
Reward $w$Wins for the player who moved hereWins aggregated over this outcome
BranchingNumber of legal movesNumber of possible random outcomes

3.2 Open Loop MCTS (Implicit Chance)

A simpler approach: instead of storing states in the tree, the tree only stores action sequences. During tree_policy, when traversing an action edge, the game engine resolves the randomness on the fly. The MCTS node doesn't cache the resulting state; it just accumulates the average rewards of all the random outcomes generated over many visits.

class MCTSNode: def __init__(self, parent, action): self.action = action # we only store the action, not the state! # ... # During traversal: state = state.apply(node.action) # samples random dice dynamically!
🧠 Tradeoff: Explicit vs. Open Loop

ApproachProsCons
Explicit (chance nodes)More accurate statistics per outcome; can prune low-probability branchesExplodes tree size (each chance outcome multiplies branching factor)
Open Loop (action nodes)Simple; no tree bloat; perfectly handles large outcome spacesCannot distinguish between lucky vs unlucky paths directly in the tree

Recommendation: Start with Open Loop MCTS for high-variance games like Yahtzee. Move to explicit chance nodes only if your game has a small number of critical random outcomes (like Backgammon dice).

4. Tuning $C_p$ for Stochastic Games

As discussed in Lesson 0001, stochastic games benefit from higher $C_p$. Because the same move can produce different results (due to dice), you need more exploration to average out the noise.

$C_p$ RangeUse Case
$1.0$ – $1.5$Deterministic games (Chess, Go, Checkers)
$1.5$ – $2.5$Mild randomness (Backgammon with doubling cube)
$2.5$ – $5.0$High randomness (Risk, Yahtzee, random card draws)

5. Check Your Understanding

✓ Quiz: Stochastic Games

1. What is a chance node in MCTS?

Correct! Chance nodes model events where randomness, not a player, determines the outcome.

2. In Open Loop MCTS, how does the tree handle random game states?

Correct! Open Loop MCTS prevents tree bloat and "optimism" bugs by representing action paths in the tree and re-sampling the actual state (like dice rolls) on the fly.

3. Should $C_p$ be higher or lower for stochastic games?

Correct! Randomness adds noise to each outcome, so you need more exploration (higher $C_p$) to gather enough samples to average out the variance.

4. What is the main downside of explicit chance nodes?

Correct! Each chance outcome adds one branch. If a die roll has 6 outcomes and the player has 10 legal moves, the branching factor at that level is 60 — the tree grows fast.

6. Summary

  • Stochastic games include random elements (dice, cards).
  • Chance nodes model random events as explicit tree nodes with probability-weighted children.
  • Open Loop MCTS resolves randomness on the fly during traversal without storing states in nodes, naturally averaging out chance events.
  • Start with Open Loop MCTS for stochastic games; use explicit chance only for small, critical chance spaces.
  • Higher $C_p$ values help average out noise in stochastic games.
