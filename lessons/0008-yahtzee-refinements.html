Lesson 0008: Yahtzee MCTS Refinements & Expert Play

Overcoming the pitfalls of Open-Loop MCTS in stochastic games through action space stabilization and heuristic upgrades.

1. The "Moderate Player" Plateau

In Lesson 0007, our MCTS Yahtzee engine successfully played full games but plateaued around a moderate score (averaging ~160 points). Expert Yahtzee play requires averaging well over 240 points. Upon analyzing the initial implementation, three major structural flaws were preventing the MCTS from achieving expert play:

  1. Open-Loop Instability with Sorted Dice: The state representation sorted the dice array, and the KEEP action used an integer bitmask based on array indices.
  2. A Destructive Rollout Heuristic: The simulate() function actively destroyed partial straights and ignored the critical Upper Section bonus.
  3. A Logic Bug in Rules: It awarded a 100-point Yahtzee Joker bonus even if the Yahtzee category had been scratched (scored as a 0).

2. Fixing Action Instability (Semantic Robustness)

The most subtle and dangerous flaw was the combination of Open-Loop MCTS and Index-based Actions on Sorted Dice.

In an Open-Loop MCTS, the tree branches on actions (not states). A deep path in the tree represents an action sequence like: Roll → KEEP(mask 0b0001) → SCORE(Fives). Because dice rolls are stochastic, the actual dice values change wildly on every simulation traverse. If the KEEP mask targets "index 0", and the dice are always sorted, index 0 simply means "keep the lowest die."

In one simulation, keeping the lowest die might mean keeping a 1. In another, it might mean keeping a 4. The semantic meaning of the action fluctuates randomly, rendering the MCTS statistics useless.

The Fix: We changed the action space from an index-based bitmask to a value-based tuple.

Instead of `KEEP(mask)`, the action became `KEEP( (5, 5) )` (meaning: "Keep two 5s"). By representing actions via their intended values, the action retains its semantic meaning regardless of the random dice rolled in the open-loop traversal.

def apply(self, move: Tuple[str, Any]) -> YahtzeeState: kind, arg = move if kind == 'KEEP': # arg is a tuple of values to keep, e.g., (5, 5) pool = list(self.dice) new_dice = [] for val in arg: if val in pool: new_dice.append(val) pool.remove(val) # If the open-loop simulation generated dice without 5s, # it just ignores it and re-rolls everything! while len(new_dice) < 5: new_dice.append(random.randint(1, 6)) return YahtzeeState(tuple(sorted(new_dice)), ...)

3. Upgrading the Rollout Heuristic

The rollout heuristic (the `simulate` function) provides the terminal value for an unexpanded node. If the heuristic is terrible, the MCTS is fed garbage data and will often perform worse than the heuristic itself.

The original heuristic kept the "most common" die. This meant if it rolled [2, 3, 4, 5, 5], it would keep the pair of 5s and destroy the Small Straight! Furthermore, it ignored the 35-point Upper Section bonus.

We completely replaced it with a smarter, human-like ruleset:

  • Protect Straights: If the Straight categories are open, scan the dice for 3-die or 4-die consecutive sequences. Prioritize keeping those over pairs.
  • Secure the Upper Bonus: If you roll 3-of-a-kind (e.g., three 6s), immediately score them in the corresponding upper section (Sixes) to lock in the 35-point bonus trajectory.
  • Smart Sacrifices: If forced to score 0 points, sacrifice categories in a specific order (Yahtzee → 1s → 2s → 3s), rather than arbitrarily burning a highly valuable category like Chance.
Impact: The pure heuristic average (without MCTS) jumped from ~179 points to ~191 points. By giving the MCTS stable, reliable terminal rewards, the MCTS baseline skyrocketed.

4. Correcting the Rules Engine

Finally, a minor bug awarded the Yahtzee joker bonus incorrectly. Standard Yahtzee rules state you only get the 100-point bonus if your Yahtzee box was scored as 50. If you scored it as 0 ("scratched"), you forfeit all future Yahtzee bonuses. We simply changed self.scores[11] >= 0 to self.scores[11] > 0.

Summary

By stabilizing the stochastic action space and providing a highly intelligent rollout policy, the Yahtzee MCTS overcomes the standard pitfalls of Open-Loop architectures. While true expert-level play in highly stochastic games often requires transitioning to Closed-Loop (ExpectiMax) MCTS to handle chance nodes perfectly, these refinements drastically bridge the gap.
