Lesson 0003: Expansion and Simulation

Adding nodes to the tree and running rollouts — phases 2 and 3 of the MCTS cycle.

1. The Flow After Selection

Recall from Lesson 0002 that tree_policy returns a node at the frontier — either a node that isn't fully expanded (has untried moves) or a terminal node. What happens next depends on which case we're in:

tree_policy result
┣─ Not fully expanded → Expansion: add one new child node
┛─ Terminal → Skip Expansion (no moves to make)
# Inside mcts_iteration (from Lesson 0002): node = tree_policy(root, Cp) # Selection phase if not node.is_terminal: node = expand(node) # Expansion phase (NEW) reward = simulate(node) # Simulation phase (NEW)

2. Expansion: Growing the Tree

Expansion is deceptively simple: take one untried move from the current node, apply it to produce a new game state, wrap that state in a new MCTSNode, link it as a child, and return it.

def expand(node): """Pick one untried move, create a child node, and return it.""" # Pop one untried move (order matters — first legal move, first tried) move = node.untried_moves.pop() # Apply the move to get a new state new_state = node.state.apply(move) # Create the child node, link it to parent child = MCTSNode(new_state, parent=node) node.children.append(child) return child
💡 Design choice: one child per call. We expand exactly one child per iteration, not all of them. This is the standard approach — it keeps each MCTS iteration cheap and lets UCT guide which branches get deepened first. Over many iterations, all moves eventually get expanded.

The state.apply(move) method is game-specific. In Chess it would make a move on the board. In Risk it would resolve a combat and advance the game state. For stochastic games (games with random elements), apply must also handle chance events — more on this in a future lesson.

3. Simulation / Rollout: The Monte Carlo Step

Simulation (also called rollout) is where the "Monte Carlo" name comes from. From the node returned by Expansion (or directly from Selection if the node was terminal), we play a full game to completion using a fast, simple policy — typically random moves. The result is a single reward signal: $+1$ for a win, $0$ for a loss, $+0.5$ for a draw.

def simulate(node): """Run a random rollout from this node's state to a terminal. Returns the reward from the perspective of the player who made the move that led to this node.""" state = node.state.clone() # don't mutate the tree's state while not state.is_terminal(): move = random_choice(state.legal_moves()) state.apply(move) return state.reward() # +1, 0, or 0.5 from current player's perspective

3.1 Why Random Works

Intuitively, random play seems useless — how can a random game tell us anything about a good move? The key insight is that we're not relying on one simulation but on the aggregate of many. If move A leads to a win in 65% of random rollouts and move B wins only 35%, then A is genuinely the better move — even though each individual rollout was random chaos.

🧠 Tradeoff: Rollout Policy Quality vs. Speed

The surprising result from the literature: uniform random is often good enough for MCTS to outperform hand-crafted AIs, because the statistical aggregation across thousands of rollouts washes out the noise. This is what made the original MCTS (before neural nets) so powerful.

3.2 Cloning vs. Applying

Notice that simulate calls state.clone() before making moves. This is critical: the rollout mutates the state (applying random moves), but we must not mutate the state stored inside our MCTSNode tree. The clone creates an independent copy for the rollout.

⚠️ Watch out! If your game state is expensive to clone (e.g., a large board with many pieces), consider a copy-on-write or undo pattern: apply a move for the rollout, then undo it. This avoids cloning overhead at the cost of more careful code.

4. Whose Perspective Is the Reward From?

A subtle but critical question: when we win a simulation, who gets credit? The reward must be from the perspective of the player who made the move that led to the current node, not from the perspective of the simulator.

For a two-player zero-sum game (Chess, Go), a win for the current player is $+1$ for them and $0$ for the opponent. During Backpropagation, we alternate the reward sign as we move up the tree — because what's good for the current player is bad for the parent. We'll cover this in Lesson 0004.

But there's another approach that simplifies things enormously: no-perspective rewards, where you always backpropagate the raw result. This works because the sign flip happens at each level of the tree naturally — you just negate the reward on each step up. We'll see the code for both approaches in Lesson 0004.

Let's trace through one iteration on a hypothetical game tree:

UCT selects A (exploitation 0.75 + exploration ...). A has one untried move remaining.

💡 Note: After expansion, we simulate from the new child C, not from A. This is the standard pattern — the rollout starts from the state we just expanded to.

6. Check Your Understanding

✓ Quiz: Expansion and Simulation

1. How many children does expand() create per call?

Correct! Standard MCTS expands exactly one child per iteration. This keeps iterations cheap and lets UCT control which branches are deepened.

2. Why does simulate() clone the state before running?

Correct! The rollout mutates the game state (applying random moves). If we didn't clone, we'd corrupt the node's state, breaking the tree for all future iterations.

3. A rollout using uniform random moves wins 55% of the time for move X. Is X likely the best move?

Correct! The aggregate of many random rollouts provides a useful signal. If move X wins 55% vs. the alternatives at 40%, it's genuinely stronger — the statistical noise washes out with enough samples.

4. What happens after tree_policy returns a terminal node?

Correct! No moves to expand from a terminal state. The simulation just returns the known terminal reward — no random rollout needed.

7. Summary

  • Expansion pops one untried move, applies it to create a new MCTSNode, and links it as a child.
  • Simulation clones the state, runs random moves to a terminal, and returns a reward ($+1$, $0$, $+0.5$).
  • Random rollouts work because aggregation across thousands of samples beats the noise.
  • The reward must be from the perspective of the player who made the move to this node.
  • Cloning (or undo) is essential to avoid corrupting tree state.
