Lesson 0010: Implementing ExpectiMax MCTS

A code walkthrough transitioning our Yahtzee engine from Open-Loop to Closed-Loop ExpectiMax MCTS.

1. Splitting the Node Class

In standard MCTS, every node is just a node. In ExpectiMax MCTS, we need to explicitly represent the alternating turns between the Player and the Environment. We split MCTSNode into two distinct classes:

class ChanceNode: def __init__(self, action: Tuple[str, Any], parent: 'DecisionNode'): self.action = action self.parent = parent self.visits = 0 self.total_reward = 0.0 # Children are specific dice outcomes, keyed by YahtzeeState self.children: Dict[YahtzeeState, 'DecisionNode'] = {} class DecisionNode: def __init__(self, state: YahtzeeState, parent: Optional[ChanceNode] = None): self.state = state self.parent = parent self.visits = 0 self.total_reward = 0.0 self.untried_moves = state.legal_moves() # Children are player actions, keyed by the action tuple self.children: Dict[Tuple[str, Any], ChanceNode] = {}

Notice how the references alternate: a Decision Node holds Chance Nodes as children, and a Chance Node holds Decision Nodes as children.

2. State Hashing (Building a DAG)

To ensure that the tree accurately learns from chance outcomes, identical dice rolls need to merge into the same state. If you roll [1,2,3,4,5] across three different simulations from the same action, they should all update the statistics of a single DecisionNode.

We achieve this by making YahtzeeState hashable so it can be used as a dictionary key in ChanceNode.children:

def __key(self): return (self.dice, self.rolls_left, tuple(self.scores)) def __eq__(self, other): return self.__key() == other.__key() def __hash__(self): return hash(self.__key())

3. Updating the MCTS Loop

The core logic of the mcts_search tree policy has to handle traversing both node types.

Step 3a: Navigating the Decision Node

At a Decision Node, the algorithm chooses a player action. It expands an untried action or selects the best explored action using the standard UCB formula.

# Inside mcts_search() # Select best action via UCB for action, chance_child in node.children.items(): exploitation = chance_child.total_reward / chance_child.visits exploration = Cp * math.sqrt(math.log(node.visits) / chance_child.visits) score = exploitation + exploration # ... track best_chance_node ...

Step 3b: Navigating the Chance Node

Once an action is selected, we are at a ChanceNode. We do not use UCB here. Instead, we ask the environment to give us a random outcome (roll the dice), and we see if we've encountered that exact outcome before.

# Sample the environment using the chosen action next_state = node.state.apply(best_action) # If we've never seen this exact dice outcome from this action, expand it! if next_state not in best_chance_node.children: child_decision = DecisionNode(next_state, parent=best_chance_node) best_chance_node.children[next_state] = child_decision node = child_decision break # Leave the tree policy loop and start the rollout! else: # We've seen these dice before. Traverse deeper! node = best_chance_node.children[next_state]

Step 3c: Backpropagation

The reward from the rollout bubbles back up the tree. Because of the alternating node types, we step through two levels at a time, incrementing visits and rewards on both the Decision Node and its parent Chance Node.

curr = node while curr is not None: curr.visits += 1 curr.total_reward += reward curr = curr.parent # Move up to ChanceNode if curr is not None: curr.visits += 1 curr.total_reward += reward curr = curr.parent # Move up to DecisionNode

4. Conclusion

By transforming the MCTS from an Open-Loop action sequence string into a Closed-Loop ExpectiMax state tree, the AI can finally formulate reactive policies. It correctly learns that an action's value isn't just one optimistic path, but the true expected value distributed across all possible dice rolls.

5. The 50,000 Iteration Benchmark (The Curse of Dimensionality)

With the architecture fixed, we benchmarked the new ExpectiMax MCTS. The results revealed a second, even deeper lesson about applying MCTS to stochastic environments:

  • 0-ply Search (Pure Heuristic): ~191.3 points
  • ExpectiMax MCTS (500 iterations): ~152.2 points
  • ExpectiMax MCTS (50,000 iterations): ~184.2 points

Despite explicitly modeling chance nodes, the MCTS still performs worse than the pure rollout heuristic!

Why? Premature Exploitation & High Variance. In Yahtzee, a KEEP action has up to 252 possible dice outcomes. To calculate a statistically accurate expected value for a single action, you need hundreds of rollouts. Across 45 possible actions, the tree is massively wide.

With only 500 iterations, the tree starves. A terrible action might get a lucky 280-point rollout. Because our exploration constant (Cp = 10.0) is mathematically dwarfed by Yahtzee's 200+ point variance, the UCB formula fails to force exploration. The MCTS becomes trapped by the very first lucky rollout it sees.

Even at 50,000 iterations, the tree struggles to fully overcome the noise, climbing to 184.2 points but still failing to beat the generalized expected-value logic built into the human-designed heuristic.

The Final Takeaway: ExpectiMax MCTS is structurally correct for Markov Decision Processes, but standard MCTS is fundamentally ill-suited for extremely wide chance nodes. To master Yahtzee, you must augment the tree with Progressive Widening (artificially limiting chance outcomes until a node is heavily visited), use a much higher Cp combined with millions of iterations, or abandon MCTS entirely in favor of Value Iteration / Dynamic Programming.
